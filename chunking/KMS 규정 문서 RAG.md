해당 HTML 은 기존 상담원이 대출 상품을 추천하기 위해 쓰던 표 

상담용 의사결정 테이블 
즉 상담원이 고객정보를 입력하면 -> 바로 대출가능/ 조건/ 한도 등을 판단하는 역할이였음

```
입력값 -> 판단조건 -> 결과 

상담로직으로 해석해야되는거 같음
```

~~~
즉 고객의 상태 = { 
	지역,
	자금용도,
	주택보유수,
	상애최초 여부 
	....
}

상태 -> 결과 

결과 -> 상담 멘트 / 승인 여부 를 판단해야함.
~~~

일반적인 RAG 라면 

~~~ 
질문 -> 관련 문서 검색 -> 요약 답변 
~~~

하지만 지금 목표는 
~~~
질문 → 상담 유형 분류 → 필요한 조건값 추출 → 판정표 row 검색 → 예외/FAQ 확인 → 상담 답변
~~~

~~~
LV1 = 상담 주제/탭/업무 상황
LV2 = 상담원이 눌러보는 답변 블록
표 = 상담 판정 매트릭스
FAQ/TIP = 상담 중 예외질문 대응용 보조 카드
[확인]/[공문] = 상담원이 필요시 클릭하는 내부 확인 버튼
~~~

으로 판단해봐야겠음 

~~~
LV1 안에 포함된 것만 기본으로 사용
LV1 밖으로는 자동 확장하지 않음
중복 청크는 canonical로만 관리
~~~

LV1 의 관계를 연결하는거 보다는 상담원이 보는 순서를 보존하는게 중요. 
LV1 은 지식구조가 아니라 UI 구조였음. 

# 질의 입력 

~~~
"서울에 집 사려고 하는데 대출 얼마나 나와?"
~~~

# 목표 

상담원이 통화/채팅 하면서 이 문서를 쓸텐데, 답변은 컨닝 페이퍼 식으로 나와야함 

정확도가 아주 높아야하니, 추론보다는 실제 데이터 기반으로 평가하는 방식이 나아 보임 



~~~
input: 서울에 집 사려는데 주담대 얼마나 나와?

	→ 상담 의도 분류  
	→ 필요한 슬롯 추출  
	→ 부족한 슬롯 확인 질문  
	→ 판정표/FAQ 조회  
	→ eligibility + limit 결과 생성

output: 서울은 규제지역 기준으로 봐야 합니다. 한도 판정을 위해 주택보유수, 주택가격, 생애최초 또는 서민실수요자 해당 여부가 필요합니다.
~~~

~~~
HTML/PDF 원문
→ HTML 정리 / DOM id 부여
→ 마커, 제목, 본문, 표 추출
→ 표 JSON 평탄화
→ LLM으로 ontology/rule 정보 추출
→ semantic chunk 생성
→ embedding 생성
→ Milvus rag_chunks 저장
[질의 파이프라인]

상담사 질문
→ query ontology 추출
   domain, product, anchor_type, rule_function, effective_date, keywords
→ query embedding 생성
→ Milvus vector search
→ reranker 재정렬
→ product/rule_function boost
→ 관련 chunk 반환
~~~
