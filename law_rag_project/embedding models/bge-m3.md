
## bge-m3 란

input 을 벡터로 임베딩해주는 모델 
이 모델은 한 input 기반 여러 검색 신호를 동시에 만들수 있다. 

- dense vector 
	-  문장의 의미를 압축한 연속 벡터 비슷한 의미이면 단어가 정확히 안 맞아도 가까워질 수 있음

- sparse vector 
	- 단어별 중요도를 살린 lexical signal 즉 조문에 실제로 들어 있는 단어와 질의 단어가 얼마나 맞는지 강하게 반영함 기존에 있던 BM25 계열 검색과 유사 

- multi-vector 
	- 토큰 단위 표현을 여러개 유지하는 방식 즉 조문 전체를 1개 벡터로 뭉개지 않고 토큰 표현들을 다 살려주는 쪽 

실제 학습에 사용되는 json 일부를 가지고 해당 데이터가 어떻게 embedding 되는지 알아보자

```json
{  
  "article_uid": "law::main::18",  
  "full_text": "제18조(유전자변형식품등의 안전성 심사 등) [제목개정 2016. 2. 3.] ① 유전자변형식품등을 식용(食用)으로 수입ㆍ개발ㆍ생산하는 자는 최초로 유전자변형식품등을 수입하는 경우 등 대통령령으로 정하는 경우에는 식품의약품안전처장에게 해당 식품등에 대한 안전성 심사를 받아야 한다. ② 식품의약품안전처장은 제1항에 따른 유전자변형식품등의 안전성 심사를 위하여 식품의약품안전처에 유전자변형식품등 안전성심사위원회(이하 “안전성심사위원회”라 한다)를 둔다. ③ 안전성심사위원회는 위원장 1명을 포함한 20명 이내의 위원으로 구성한다. 이 경우 공무원이 아닌 위원이 전체 위원의 과반수가 되도록 하여야 한다. ④ 안전성심사위원회의 위원은 유전자변형식품등에 관한 학식과 경험이 풍부한 사람으로서 다음 각 호의 어느 하나에 해당하는 사람 중에서 식품의약품안전처장이 위촉하거나 임명한다. 1. 유전자변형식품 관련 학회 또는 「고등교육법」 제2조제1호 및 제2호에 따른 대학 또는 산업대학의 추천을 받은 사람 2. 「비영리민간단체 지원법」 제2조에 따른 비영리민간단체의 추천을 받은 사람 3. 식품위생 관계 공무원 ⑤ 안전성심사위원회의 위원장은 위원 중에서 호선한다. ⑥ 위원의 임기는 2년으로 한다. 다만, 공무원인 위원의 임기는 해당 직(職)에 재직하는 기간으로 한다. ⑦ 식품의약품안전처장은 거짓이나 그 밖의 부정한 방법으로 제1항에 따른 안전성 심사를 받은 자에 대하여 그 심사에 따른 안전성 승인을 취소하여야 한다. ⑧ 제2항부터 제6항까지에서 규정한 사항 외에 안전성심사위원회의 구성ㆍ기능ㆍ운영에 필요한 사항은 대통령령으로 정한다. ⑨ 제1항에 따른 안전성 심사의 대상, 안전성 심사를 위한 자료제출의 범위 및 심사절차 등에 관하여는 식품의약품안전처장이 정하여 고시한다."
```

### 1. bge/m3 의 내부 tokenizer 

input으로 들어온 full_text 를 tokenzier 가 잘게 쪼갬. 
위 예시로 쪼갠다면 

- 제18조
- 유전자
- 변형
- 수입 
- 20명 

이런식으로 내부토큰 단위로 쪼개짐 


### 2. 각 토큰을 숫자 벡터로 바꿈 

기본적으로 한 토큰당 1024차원 벡터로 바꿈

1차적으로 진행했던 하나의 토큰을 내부적인 모델 사전에 있는 정수 ID 로 바꿈 

```
["영업", "허가", "식품"]  
→ [18452, 9271, 33810]
```

2차적으로는 숫자 ID를 임베딩 테이블에서 꺼냄 

모델 안에는 embedding matrix 가 있음 

row 는 토큰 ID 
column 은 벡터 차원 

"허가" 이라는 토큰은 embedding matrix 의 9271 번째 행을 꺼내는 과정을 거침 

```
token_id = 9271  
→ embedding_matrix[9271]  
→ [0.12, -0.04, 0.88, ..., 0.03]
```

embedding matrix 의 벡터값은 BAAI측에서 사전학습된 벡터값들이다 
필요하다면 fine-tuning 할수있지만 일단 pretrained bge-m3 로 성능 확인 뒤 
reranker , query rewrite 조정해보고 부족하면 그때 진행해도 늦지 않다 

즉 이미 trained 된 모델은 "허가" 와 "승인" 이런 토큰들이 이미 비슷한 위치에 있다는거고
필요하다면 fine-tuning 을 통해 해당 토큰들의 위치를 조정할수있다. 
### 3. transformer 

이렇게 임베딩 테이블에서 꺼낸 직후의 벡터들은 문맥을 반영하지 않는다. 
만약 tokenizer 가 "허가" 라는 토큰 ID 를 가지고있다고 해보자. 

- 건축 허가
- 영업 허가
- 수입 허가

해당 문장들의 "허가" 는 다 같은 기본 벡터로 시작할거다. 
허가라는 기본벡터만 본다면 "허가" 라는 키워드가 들어왔을때
이 허가가 건축허가인지, 영업 허가인지 수입허가 인지 구분을 못한다. 

#### 위치정보 

문맥을 파악하려고 token embedding 벡터에 + position embedding 을 구별해준다. 

#### 문맥 

transformer 는 그래서 "허가" 토큰을 만들때 주변 단어를 참조한다. 

	영업 허가를 받아야 한다.

여기서 "허가" 는 주변의 
- 영업
- 받아야
- 한다

같은 단어를 보고 해당 허가를 "영업 관련 허가" 쪽으로 벡터를 조정한다. 

문장의 "허가" 라는 단어는 tokenizer 를 거쳐 토큰 ID 가 부여되고,  
embedding table 을 거쳐서 기본벡터로 반환되지만, 문맥을 고려하려고 

1. embbeding table + position 

2. transformer의 문맥 반영 (self-attention) 

을 거쳐 최종 벡터가 된다. 즉 같은 허가여도 다 다른 벡터값을 갖는다. 

### 조문 전체 임베딩 

한 조문은 N개의 토큰을 가진다. 
각 토큰은 위의 과정을 거쳐 1024차원 벡터를 가진다. 

조문 단위로 chunk 했다는뜻은, 질의가 들어올때 해당 질의가 찾는 가장 근접한 
조문을 뽑는거다. 

query -> embedding -> 1024차원 벡터 -> 가장 유사한 조문벡터를 뽑아야된다

즉 이 시점에서는 
N개의 1024차원 벡터를 압축해서 한 조문에 대한 1024차원 벡터를 만들어야한다. 

이걸 **pooling** 이라고 한다. 

bge-m3 모델의 dense 벡터는 기본적으로 **CLS** 를 사용해 조문 벡터를 만든다. 

여기서 **CLS** 란 한 조문의 **문장 대표자** 역할을 하라고 넣어둔 특수 토큰이고, 
Transformer 를 지나면서 다른 토큰 정보를 빨아들여 최종적으로 조문 요약 벡터처럼
쓰인다. 

다음 입력이 주어졌다고 해보자 

```
제36조(영업의 허가) 식품제조·가공업을 하려는 자는 ...
```

처음에는 **CLS** 도 그냥 기본 벡터이다 

```
[CLS] = [0.1, 0.1, 0.1]
제36조 = [0.2, 0.0, 0.1]
영업   = [0.8, 0.9, 0.1]
허가   = [0.9, 0.8, 0.2]
식품   = [0.7, 0.5, 0.6]
제조   = [0.6, 0.4, 0.5]
```

이런식으로 말이다. 

Transformer 단계에서 **CLS** 는 다른 토큰을 본다. 

**self attention** 단계에서 전체의 토큰들과의 관계를 계산한뒤, 해당 토큰의 정보를 가중합해서 
자신의 벡터를 업데이트 한다. 

bge/m3 의 transformer 단계는 약 24층으로 구성되는데, 각 transformer 층을 통과할때마다 점점 정교해진 벡터를 만들게 된다. 


# sparse 

sparse 는 **특정 키워드 중심으로 매칭한다** 라는 측면에서 **BM25** 와 유사하지만, 
**transformer** 가 만든 각 토큰의 **hidden state** 을 이용해서 이 토큰이 얼마나 중요한지
가중치를 학습해 뽑는게 특징이다. 

조금 더 설명하자면 

**BM25** 는 주로 단어의 빈도, 문서 빈도같이 해당 단어가 얼마나 나왔는지를 통해 해당 단어의 중요도를 평가한다. 

**BGE-M3 sparse** 는 입력 문장을 넣었을때 transformer 를 거친뒤, 문맥이 반영된 이후 겹치는 토큰들의 가중치를 이용해 **lexical score** 를 계산한다. 

BM25는 기본적으로 
- TF : 그 단어가 문서에 얼마나 나왔는지
- IDF: 그 단어가 전체 문서들에게서 얼마나 희귀한지 
- 길이 정규화: 문서가 너무 길면 점수를 조정 

BGE-M3 sparse는 
- 문장을 encoder에 넣고 hidden state 를 만든다 **transformer 과정**
- 해당 hidden state 을 가지고 토큰의 weight 을 계산한다 
- query 가 들어왔을때 해당 쿼리와 겹치는 토큰들의 **weight** 을 이용해서 점수를 계산한다.
