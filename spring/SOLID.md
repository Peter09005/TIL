

- SRP

	하나의 클래스는 한개의 책임을 가진다
	중요한 기준은 변경임 --> 해당 클래스를 변경했을때 파급효과가 적다면 단일 책임원칙을 잘 따른 것 

- OCP

	확장에는 열려있으면서 변경에는 닫혀있어야됨. 
	기존코드를 변경하는게 아닌 역할을 구현하면 기존 코드의 변경없이 확장을 할수있다 

- LSP

	다형성에서 하위클래스는 인터페이스 규약을 다 지켜야함

- ISP

	특정 클라이언트를 위한 인터페이스 여러개가 범용 인터페이스보다 낫다
	ex) 자동차 인터페이스 -> 운전 , 정비 인터페이스로 분리


- DIP 

	추상화에 의존, 구체화의 의존 X
	구현 클래스에 의존하지 말고 역할에만 집중해야함 


DIP , OCP --> 스프링의 의존성 주입 기능으로 해결 가능하다. 

Spring -> DI 컨테이너 
클라이언트 코드 변경 없이 기능 확장 가능
